# Changelog

## 0.4.2 - 2026-07-24

- Fixed consumer typechecking: the `types` condition now resolves to a shipped declaration tree (`types/**/*.d.ts`) instead of raw `src/*.ts`, so a consumer's own compiler flags never typecheck this package's implementation. A consumer on `--strict --exactOptionalPropertyTypes` saw 73 errors (85 with `--noUncheckedIndexedAccess`) from inside `src/core.ts` and friends on 0.4.1; it now sees 0.
- The declaration tree is committed (regenerated by `bun run build:types`) because a scriptless exact-Git install can only ship what the Git tree already contains. `bun run verify:shipped-types` re-emits declarations and fails on any drift from `src/`; it is wired into `verify:release` and CI (which also fails on an uncommitted `types/` diff).
- Fixed consumer installs: removed the `@types/node` / `@types/pg` peer dependencies. A consumer pinning `@types/node@^18.19.0` could not install 0.4.1 at all (`npm error ERESOLVE ... peer @types/node@">=20" from @hasna/identities`). Type packages must never hard-block an install — not even as optional peers, which npm still rejects when a present version is out of range.
- `@types/pg` is now an ordinary wide-range dependency (`^8.11.10`) because the shipped `pg-store` declarations reference `pg` types; on a conflicting consumer pin it nests instead of failing resolution. `@types/node` and `@types/bun` are not imposed on consumers at all.
- `resolveCloudHttpConfig` / `resolveStorageTransport` now accept a structural `EnvironmentVariables` (`Record<string, string | undefined>`) instead of the Node ambient `ProcessEnv` type, so no shipped declaration needs `@types/node`. `process.env` remains assignable, so this is source-compatible.
- Widened `bun run verify:package-consumers`: every consumer probe now typechecks under `--strict` and under `--strict --exactOptionalPropertyTypes --noUncheckedIndexedAccess` (Bundler and NodeNext), asserts no `types` condition points into `src/`, asserts no type package is a peer, and adds a consumer case that pins `@types/node@^18.19.0` + `@types/pg@^8.6.6` and requires the install to succeed.

## 0.4.1 - 2026-07-24

- Fixed scriptless exact-Git installs: `bun add github:hasna/identities#<sha>` previously produced an unusable package because the tree has no `dist` and the install-time hook was blocked as an untrusted lifecycle script.
- Added a `bun` export condition, source `types`, and `src/**/*.ts` (minus tests) to the published files so Bun and TypeScript resolve the shipped source while Node keeps importing the built `dist` output.
- Pointed the `identities`, `identities-mcp`, and `identities-serve` bins at the executable shipped sources so they run from a Git install as well as a registry install.
- Removed the `postinstall` directory hook; the store already creates its data directory on demand with `0700` permissions, so the hook was redundant and only widened those permissions.
- Declared `@types/node` and `@types/pg` as wide-range peer dependencies (never exact runtime pins) so consumers typechecking the shipped source dedupe them instead of getting a second copy; `@types/bun` is not imposed on consumers at all.
- Added `bun run verify:package-consumers`, wired into `verify:release` and CI, which packs the tarball and proves every export, type condition, and bin works for a Bun consumer, an npm/Node consumer, and (with `--spec`) an exact-SHA Git consumer, with no trusted lifecycle scripts.
